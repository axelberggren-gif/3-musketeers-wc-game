01

The merge gate — how a PR ships

main is branch-protected: direct pushes are rejected and three checks must be green before anything merges. An approval alone never ships code — it only removes the human-merge step.

PR → main pipeline

open PR
feat/ax/… Conventional-Commits title required
→
check 1
ci lint · typecheck · test · build
→
check 2
claude-review / review AI bot approves or requests changes
→
check 3
pr-title-lint / lint title = valid commit subject
→
auto
squash-merge → Vercel promotes to prod
All three are required status checks. auto-merge.yml flips on GitHub auto-merge once a trusted author (or the bot) approves — but branch protection holds the merge until every check passes. Squash is the only merge strategy, so the PR title becomes the commit on main.
02

Continuous integration & supply-chain security

Automated checks on every pull request and push to main — the build loop, plus the deterministic supply-chain and security-scanning layer added June 2026.

CI Build pipeline

One job runs the full loop in order: npm ci → lint → typecheck → test → build. Any step failing fails the check and blocks merge.

The build step deliberately runs with placeholder Supabase env and an empty Sentry DSN — so CI exercises the "Sentry no-ops when unconfigured" path on every run.

.github/workflows/ci.yml

CI PR title lint

Enforces Conventional Commits on the PR title via action-semantic-pull-request. Allowed types: feat, fix, chore, docs, refactor, perf, ci, revert, style. Subject must start lowercase.

Matters because squash-merge turns the PR title into the permanent commit message on main.

.github/workflows/pr-title-lint.yml

CI Type safety

tsc --noEmit over the whole tree. Supabase queries are fully typed via a generated Database namespace threaded through every client, so a schema change that isn't reflected in types.ts surfaces as a type error.

npm run typechecklib/supabase/types.ts

CI Auto-merge orchestration

On an approving review from a trusted author, enables GitHub native auto-merge (squash). Idempotent and gated on author_association ∈ {OWNER, MEMBER, COLLABORATOR} so a drive-by fork can never trigger a merge.

.github/workflows/auto-merge.yml

CI SHA-pinned actions

Every uses: across all eleven workflows is pinned to a full 40-char commit SHA (with a # v4 comment), never a mutable tag. Closes the supply-chain hole behind the 2025 tj-actions and 2026 trivy-action tag-hijack attacks that exfiltrated CI secrets.

.github/workflows/*

CI Dependabot

Weekly update PRs for both npm and github-actions — patches vulnerable deps and bumps the pinned action SHAs so pinning never rots. PR titles carry Conventional-Commits prefixes so they pass pr-title-lint.

.github/dependabot.yml

CI Dependency audit

npm audit --audit-level=high fails the check on any high/critical advisory in the dependency tree. Free + private-repo-friendly (GitHub's dependency-review needs paid Advanced Security on private repos). Dependabot does the upgrades; this is the gate.

.github/workflows/dependency-audit.yml

CI Semgrep (SAST)

Semgrep OSS scans JS/TS for injection, XSS, and unsafe data flow — a deterministic backstop to the (non-deterministic) AI review bot. Free + private-safe (scans locally, uploads nothing); on PRs it gates only on newly-introduced findings via a baseline diff.

.github/workflows/semgrep.yml

CI Secret scanning

gitleaks scans every push and PR for committed credentials — defense in depth over GitHub-native secret scanning + push protection. Catches the documented AI failure mode of hardcoding the Supabase service-role key or CRON_SECRET.

.github/workflows/secret-scan.yml

CI Changelog guard

Any PR touching runtime code (app/, components/, lib/, supabase/) must update CHANGELOG.md, and the added lines must reference the PR number (#NN). Turns the worst-drifting convention into a machine check; the skip-changelog label is the escape hatch. Checkout-free — pure GitHub API.

.github/workflows/changelog-guard.yml

CI ESLint core correctness

js.configs.recommended runs ahead of the Next configs (eslint-config-next alone leaves every eslint:recommended rule off), with no-unreachable explicitly re-enabled for TS files — the typescript-eslint overlay disables it, which is how a shipped unreachable-code bug once passed lint. Empty catch {} blocks now need an intent comment.

eslint.config.mjs
03

AI review agent

A required check backed by anthropics/claude-code-action. It reads the project's canon (AGENTS.md + per-directory CLAUDE.md) before the diff, then approves or requests changes. Its verdict gates the merge.

AI review Verdict-as-gate

A separate "Gate" step reads the bot's most-recent formal review state. APPROVED passes; CHANGES_REQUESTED — or a missing / comment-only review — fails the check.

Needed because GitHub excludes bot reviews from branch-protection's "required approvals" count, so the workflow's own exit code is what actually gates auto-merge.

.github/workflows/claude-review.yml

AI review Blocks on correctness & security

Requests changes for logic errors, broken contracts, race conditions, unhandled error branches that crash prod, and security issues: XSS, SQL injection, command injection, secrets exposure, and RLS leaks.

AI review Enforces project invariants

Explicitly checks the load-bearing rules: points-sync, append-only migrations, cron-secret auth, service-role guard, RLS-first pick visibility, the server-action result shape, idempotent scoring, and the AI co-author trailer on every commit.

AI review Demands test coverage

New logic in lib/scoring, predictions, banter, auth, or football-data needs a test or a written justification. Pure UI / copy / docs PRs are exempt. Won't block on naming, comment density, or style nits the codebase doesn't already follow.

AI review Trust boundary

Only runs on non-draft PRs from trusted authors. Authenticates against the maintainer's Claude subscription via OAuth token (not a metered API key), and is granted a narrow tool allowlist so it can actually post the review instead of silently drafting one.

AI review Inline + summary

Posts one inline comment per blocker anchored to the offending line, then a single formal review summarising findings — the same artifact the gate step reads back.

CI Post-merge review follow-up

The review gate can lose the race to the merge (it happened twice — verdicts posted after the PR landed, and the blockers vanished). When a CHANGES_REQUESTED review arrives on an already-merged PR, this workflow files a prio:P1 issue carrying the full review, so post-merge blockers survive until triaged. Deduped by title; the review body is passed via env, never interpolated into shell.

.github/workflows/review-followup.yml
04

Local & commit-time guards

Fast feedback before code ever leaves the machine, plus a reusable verify subagent.

git hook Husky pre-commit

Runs npm run lint && npm run typecheck && npm test on every commit (the vitest suite costs ~1.4 s), catching the three cheapest failures before they reach CI. Wired up automatically via the prepare: husky install script.

.husky/pre-commit

git hook Commit-message gate

A commit-msg hook validates the Conventional Commits subject and requires the Co-authored-by AI trailer on every commit — the two AGENTS.md rules that drifted most under convention alone. Merge / revert / fixup messages pass through. POSIX sh, no deps.

.husky/commit-msg

convention Verifier subagent

A dedicated Claude Code subagent that runs lint → typecheck → test → build, stops at the first failure, and reports a ≤40-line actionable excerpt — keeping the parent agent's context small. It can't edit code or propose fixes; its only output is a pass/fail report.

.claude/agents/verifier.md

convention Git-identity protection

Agents are forbidden from running git config user.name/email — the human sets it once per clone. Overwriting it would impersonate a teammate. The session ritual verifies the identity is a human name, not "Claude".

Test suites

vitest runs in CI. The suites concentrate on the highest-stakes pure logic — scoring, locks, and the analytics eval schema.

CI Points-sync drift (both directions)

rules.test.ts pins every value in the POINTS object; points-sync.test.ts covers the SQL side — it parses the migrations in apply order (last create or replace wins, drops remove) and asserts the live points_*() constants equal POINTS in both directions, plus points_bracket_slot()'s CASE arms. A migration that skews a point value now goes red instead of silently mis-scoring the league.

06

Codebase invariants

The rules that, if broken, quietly corrupt the game. Each is defended in more than one place — DB, build, tests, and the review bot — so no single missed check is catastrophic.

DB Scoring idempotency

Every write to point_awards goes through score_*() SQL functions keyed on idempotency_key — safe to re-run under the 10-min cron. A later migration (0014) added delete-stale-then-insert reconciliation so a corrected result revokes now-wrong awards.

0002_scoring.sql0014_reconcile_scoring.sql

DB RLS is the source of truth

Who can see whose picks is enforced by Row-Level Security, not app code. Picks are invisible to other users before kickoff via the *_read_after_kickoff policies. Locks are enforced in three independent layers: DB triggers, RLS, and the UI.

0001_init.sql

DB Migrations are append-only

Merged migrations are never edited — changes land as a new numbered file. Tracked as a critical safety eval case and a hard block in the review bot, machine-checkable straight from the diff.

supabase/migrations/

runtime Cron-secret auth

Every handler under app/api/cron/* verifies CRON_SECRET via the shared authorizedCron() helper, which compares with crypto.timingSafeEqual (length-mismatch short-circuits first) — never a raw ===.

lib/cron/auth.ts

runtime Service-role guard

supabaseService() bypasses RLS, so it's never called from user-facing routes without an explicit authorization check above it. A leak here would reveal locked picks — catastrophic for a predictions game.

lib/supabase/server.ts

runtime Points-sync (two-file rule)

A point value can't change in rules.ts alone — it requires a matching new migration updating the SQL function. Mirrored for FIFA_RANKINGS_2026 ↔ the teams.fifa_ranking seed. Backed by the points-sync test.

lib/scoring/rules.ts

runtime Server-action result shape

Every server action returns the discriminated union { ok: true } | { ok: false; error }, and the UI handles both branches with optimistic-update-then-rollback. Copy-paste baseline for every new picker.

MatchPickCard.tsx

DB 180-char banter cap (triple)

The chat body length limit is enforced in three aligned places: the client counter+disable, the server action's validateBody(), and a Postgres CHECK constraint. Change one, change all three.

0011_banter.sql

runtime Middleware scoping

The session-refresh proxy matcher excludes /api/cron/* and /_next/data/*, so pg_cron pings don't trigger a Supabase session refresh on every hit.

proxy.ts
07

Error tracking & privacy

Production errors and session replays land in Sentry — with privacy guardrails baked into the integration.

runtime Gated, no-op when unconfigured

The whole integration is gated on NEXT_PUBLIC_SENTRY_DSN, so local and CI builds without the env vars no-op cleanly — and CI proves it by building with an empty DSN.

instrumentation.tssentry.server.config.ts

runtime No PII in events

Sentry.setUser is only ever called with { id, username } — never email. Session replay is errors-only and masks all inputs, which covers the login email field.

runtime Structured capture helpers

Server actions surface handled errors through captureServerActionError (which flushes before the serverless function freezes); cron handlers tag exceptions with { cron: "<name>" }. Migration-drift is captured with the PG error code so the next drift is diagnosed without a DB probe.

lib/sentry/capture.ts
08

Agent product analytics

The most unusual guardrail: since AI agents are the developers here, the repo instruments the agents' own work. A PR = one agent run, the area label = the workflow, and corrections (post-merge fixes, reverts, review CHANGES_REQUESTED) are the acceptance signal. No product code, no dependencies — just Node built-ins over git + GitHub history.

Q1 · TrustedHigh completion, low correction. Agents ship clean — candidate for more autonomy.
Q2 · Finishing work nobody trusts ⚠Ships, but keeps needing fixes. Dashboards call it healthy; it isn't. Tighten quality & mine corrections into evals.
Q3 · Failing before reviewLow completion and high correction — structural. Audit tooling/permissions/context.
Q4 · Too cautious but valuableAccepted when it finishes, but finishes too rarely. Over-gated — reduce friction.

CI Live four-quadrant report

A GitHub Action recomputes the completion-vs-acceptance report weekly, on every merged PR, and on demand, then upserts a single tracking issue. It deliberately commits nothing — main is protected and the report is cheap to recompute.

.github/workflows/agent-analytics.ymlscripts/report.mjs

convention Correction → eval loop

Every correction becomes a candidate regression test. agent:evals drafts an eval stub from the correction; the committed cases under evals/cases/ are then validated by npm test — closing the loop from "agent got it wrong" to "CI catches it next time".

scripts/correction-to-eval.mjsevals/cases/*.json

convention Fail-safe by design

Scripts and the optional local hooks must never break a build, a session, or CI — gh/git failures degrade gracefully and the hook always exits 0. The dir adds no deps and touches neither the product runtime nor the DB.

analytics/CLAUDE.md
09

Process & documentation guardrails

The softer rails — templates, labels, and the "canon docs" system that puts invariants next to the code that has to honor them.

convention The CLAUDE.md canon system

A root AGENTS.md plus a per-directory CLAUDE.md in every significant area. Each is marked Canon — if reality and the doc disagree, you fix the doc in the same PR. Invariants live beside the code, not in a wiki that rots.

AGENTS.md*/CLAUDE.md

convention PR template checklist

Every PR ships a checklist: typecheck/lint/build pass, CHANGELOG updated, touched CLAUDE.md refreshed, no migration edits an existing file, and point-value changes touch both rules.ts and a new migration.

.github/pull_request_template.md

CI Label sync

Labels are declared in code (labels.yml) and synced to GitHub by a workflow on push-to-main. Type/area/priority taxonomy stays consistent and reviewable.

.github/labels.ymlsync-labels.yml

convention Structured issue intake

Blank issues are disabled; everything routes through Bug / Idea / Task forms that auto-apply the matching type:* label. Keeps the tracker triage-able.

.github/ISSUE_TEMPLATE/

convention Standing code review

A living CODE_REVIEW.md audit grades open risks P0/P1/P2 with file:line citations and records resolutions inline — e.g. the DEV_INSTANT_LOGIN prod foot-gun, removed wholesale before onboarding.

CODE_REVIEW.md

convention AI co-author trailer

Every agent-authored commit ends with a Co-authored-by: Claude trailer, making AI-assisted work queryable across history — and feeding the analytics attribution.

Defense in depth, by design. The load-bearing rules are never trusted to a single check. Points-sync is guarded by a unit test, the PR checklist, and the review bot. Append-only migrations are an eval case, a review-bot block, and a documented invariant. Pick visibility is enforced by DB triggers, RLS policies, and the UI. Any one rail can miss without the game breaking.
