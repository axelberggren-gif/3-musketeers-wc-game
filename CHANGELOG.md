### Fixed
- 2026-05-25 (`pending`) `/predict` Tournament outcome team dropdowns came up empty in production — Sentry confirmed `42703: column "teams.fifa_ranking" does not exist` because migration 0005 had never been applied to the live DB. Split the catalogue query in `app/(app)/predict/page.tsx` so the team list (core columns from 0001) stays separate from the rank lookup (0005 column); the rank fetch is allowed to fail without nuking every picker — only the dark-horse rank-sort degrades while the form remains usable. The rank-query failure is still captured to Sentry (level `warning` now) so the missing migration stays visible. Refs #47 — @ax
- 2026-05-25 (`pending`) `<CountdownBanner>` triggered React hydration mismatches on `/predict` (Sentry `JAVASCRIPT-NEXTJS-5`, replay-attached). The lazy `useState` initializer called `Date.now()` during SSR, and `Date.now()` at hydration time was several hundred ms later, so the formatted countdown text differed. `remaining` now starts as `null` (placeholder DOM is `--:--:--` on both server and client first render); a `requestAnimationFrame` inside `useEffect` schedules the first real tick, then a 1 s interval drives the countdown. Same fix applies on `/predict/bracket` since both pages use the banner. Refs #47 — @ax
- 2026-05-25 (`pending`) `/predict`'s `teams` / `players` catalogue queries silently emptied out when a migration hadn't been applied to the target DB — PostgREST returned 400 (`column "fifa_ranking" of relation "teams" does not exist`), supabase-js filled `error` while data stayed null, the `?? []` coalesce wiped the dropdowns, and users saw blank Tournament outcome pickers with no signal in the logs. `app/(app)/predict/page.tsx` now `Sentry.captureMessage`s both `teamsRes.error` and `playersRes.error` (with `pg_code` / `pg_message` / `pg_details` / `pg_hint` extras and a `feature=tournament_predictions` tag) so the next migration drift surfaces immediately. Refs #47 — @ax
- 2026-05-24 (`pending`) Every group-stage match was tagged "Group G" (and any hand-backfilled `teams.group_letter` followed suit). Root cause: `syncFixtures()` parsed `m.group` with `replace("Group ", "").slice(0, 1)`, but football-data v4 returns `"GROUP_A"`..`"GROUP_L"` (uppercase/underscore, matching the `stage` enum convention) — not the legacy `"Group A"` form. The replace was a no-op and `.slice(0, 1)` always returned `"G"`. New `parseGroupLetter()` helper accepts both formats and validates the trailing letter is A–L. Also adds migration `0010_backfill_team_group_letter.sql` with a `backfill_team_group_letters()` RPC that propagates `matches.group_letter` → `teams.group_letter` (nothing else writes that column — `seedTeams()` never had access to group info, leaving both `GroupWinnerPicker` and `score_first_eliminated` reading a NULL column). `syncFixtures()` invokes the RPC after the match upsert loop so one sync cycle self-heals both tables — @ax
- 2026-05-24 (`pending`) `redeem_league_invite()` was rejected at runtime with `42804: structure of query does not match function result type` (`Returned type citext does not match expected type text in column 2`) — once migration 0004 was finally applied to prod, every invite redemption raised this PG error. The function declared `league_slug text` but the success path returned `inv.slug` (where `leagues.slug` is `citext`). Migration `0009_redeem_league_invite_citext_cast.sql` casts the slug to `text` in the success-path SELECT. No signature change — TS callers unaffected — @ax
- 2026-05-24 (`pending`) Invite-redemption failures were silently swallowed: when `consumeInviteForUser()` returned `!ok`, the DEV_INSTANT_LOGIN sign-in flow re-routed to `/leagues` (empty Pick Your Crew state) and `/auth/callback` did the same after a magic-link round-trip. User got logged in but didn't know why they weren't in the league. `consumeInviteForUser` now `Sentry.captureMessage`'s both RPC errors and `!row.ok` responses with token prefix, user_id, and raw RPC error/row shape so we can finally diagnose what's actually rejecting. `signInWithEmail` (DEV path) returns `{ ok: false, error }` on consume failure so `LoginForm` shows it inline. `/auth/callback` now bounces through `/join/[token]` after magic-link sign-in so the single redemption code path handles both flows; `JoinPage` renders an explicit "Couldn't join" error state (with link back to /leagues) when a logged-in user's consume fails instead of confusingly showing the login form — @ax
- 2026-05-24 (`pending`) `/join/[token]` crashed in production (Sentry `JAVASCRIPT-NEXTJS-3` / `JAVASCRIPT-NEXTJS-4`) because the Server Component called `setPendingInvite(token)`, which `cookies().set(...)`s — forbidden outside Server Actions / Route Handlers in Next.js 16. The invite token already round-trips via the magic-link's `?invite=` URL param (passed to `LoginForm` → `signInWithEmail` → `emailRedirectTo`), so the cookie fallback was redundant. Removed the cookie write + the now-dead `read/clear` helpers and the matching fallback branch in `app/auth/callback/route.ts` — @ax
- 2026-05-24 (`pending`) Round 4 — Sentry event 122257777 finally pinned the root cause: `infinite recursion detected in policy for relation "league_members"`. The 0006 OR'd bootstrap clause wasn't enough because Postgres evaluates the OR'd EXISTS subquery regardless of whether the first OR branch short-circuits, and the EXISTS still references `league_members` from inside `league_members`' own policy. Migration `0008_rls_security_definer_helper.sql` adds an `is_league_member(league_id, user_id)` SECURITY DEFINER helper that runs with RLS bypass, then rewrites both `league_members_read_self_leagues` and `leagues_read_members` to route their membership check through the helper instead of EXISTS'ing back into `league_members` — @ax
- 2026-05-22 (`pending`) Round 3 of the league-creation visibility bug. Migration `0006_fix_league_members_rls.sql` wasn't enough; users still saw "League created but not visible to you" in production. Migration `0007_leagues_owner_base_case.sql` adds an `owner_id = auth.uid()` base case directly to `leagues_read_members` so a creator can read their league back regardless of how the `league_members` EXISTS subquery resolves. Also adds a `debug_auth_uid()` SQL function; the `createLeague` action now calls it (plus probes the user's own profile and league_members rows) and captures the results to Sentry as `extra` data on the same `captureMessage`, so the next time anyone hits this we'll see the PostgREST-side `auth.uid()`, whether `role=authenticated` (profile readable), and whether the user's own membership row is RLS-visible — @ax
- 2026-05-22 (`pending`) Root cause of the league-creation 404 — finally. The `league_members_read_self_leagues` RLS policy was fully self-referential: to read any row in `league_members` you needed to find another row in `league_members` for the same league, which is the very thing the policy was supposed to gate. Postgres' RLS recursion-breaker returned false, so a user couldn't see their own freshly-inserted membership row. That cascaded into `leagues_read_members`' EXISTS check returning false, so the brand-new league was invisible to its creator. Migration `0006_fix_league_members_rls.sql` adds a `user_id = auth.uid()` bootstrap clause so the policy has a non-recursive base case. The Sentry-tagged sanity check added in #24 confirmed the symptom in production — @ax
- 2026-05-22 (`pending`) League creation 404 — round 2 follow-up to #23. The previous fix used `router.push()` after a `{ ok, slug }` return, but the bug still reproduced in production. Switched `CreateLeagueForm` to the canonical `useActionState` + `<form action={action}>` pattern so the action's `redirect()` propagates through React's form action handling. Added an RLS-aware visibility sanity check inside `createLeague`: after both inserts, query the new league back through the user's RLS-aware client. If the user can't see their own league, roll both inserts back, capture to Sentry with `user_id`/`league_id`/`slug`, and return an explicit error to the form instead of letting the redirect land on a 404 — @ax
