## 2. Method & environment

The sandbox has **no Docker daemon** and **no Supabase CLI**, so the full hosted
Supabase stack can't run. It *does* have **PostgreSQL 16** locally. Because the
"entire app's calculation" lives in SQL scoring functions (+ `lib/scoring`), the
simulation drives those functions directly against local Postgres — the
**simulation is the source of truth**: real migration SQL runs over synthetic
data and the resulting `point_awards` rows are inspected.

**Platform shims** (so the app's migrations apply to plain Postgres — these mimic
objects the hosted Supabase platform provides; they do not change app logic):

- `auth` schema + `auth.users` table + `auth.uid()/role()/jwt()` (resolve from
  `request.jwt.*` GUCs, exactly like Supabase).
- Roles `anon`, `authenticated`, `service_role`, `authenticator`.
- `citext` + `pgcrypto` extensions; a `supabase_realtime` publication.

The simulation runs as the Postgres superuser, which bypasses RLS — equivalent to
the **service‑role** path the cron/admin scorers use. The round‑1/round‑2 **lock
triggers still fire** (triggers run regardless of RLS), so predictions were
written while the tournament dates were in the future, then the dates were moved
to the past to "start" the tournament.

Phases: `00` shims → `01` onboard+catalog → `02` fixtures → `03` truth+predictions
→ `05` play+score → `06` verify. (Scripts live in `/tmp/wcsim/*.sql`; reproduction
in §9.)

### 3.2 Each scorer — trigger, formula, idempotency, leaderboard bucket

Authoritative definitions are the **latest** `create or replace` after applying all
migrations in order: `0014` for match/bracket/tournament + tie‑split props, `0005`
for group‑winner/first‑eliminated, `0013` for the R32 point value.

| Category | Function | Fires when | Award rule | Idempotency key | `prediction_type` → bucket |
|---|---|---|---|---|---|
| **Group 1X2** | `score_match(match_id)` | a match is `FINISHED` w/ a `winner` | every `match_predictions.pick == matches.winner` → **3** | `match:<uid>:<match_id>` | `match` → *match_points* |
| **Bracket** | `score_bracket()` | any slot's match `FINISHED` | pick's team == the team that advanced (won the slot's match; `W` resolves to the Final) → **1/2/4/6/10/15** by stage | `bracket:<uid>:<slot>` | `bracket` → *bracket_points* |
| **Winner** | `score_tournament()` | Final `FINISHED` | `winner_team_id == champion` → **25** | `tournament:winner:<uid>` | `tournament` → *tournament_points* |
| **Runner‑up** | `score_tournament()` | Final `FINISHED` | `runner_up_team_id == finalist that lost` → **10** | `tournament:runner_up:<uid>` | `tournament` |
| **Dark horse** | `score_tournament()` | Final `FINISHED` | picked team appears in a **QF** match **and** has a non‑null `fifa_ranking` → **= that rank (1..48)** | `tournament:dark_horse:<uid>` | `tournament` |
| **Top scorer** | `score_tournament()` | Final `FINISHED` | `top_scorer_player_id` is (tied) max goals in `player_goal_log` → **15** (all ties win) | `tournament:top_scorer:<uid>` | `tournament` |
| **Player prop** | `score_tournament()` | Final `FINISHED` | `player_prop_predictions.player_id == player_prop_resolutions.player_id` for that `prop_key` → **10** | `prop:<uid>:<prop_key>` | `prop` → *prop_points* |
| **Total goals** | `score_total_goals_guess()` | Final `FINISHED` | closest `total_goals_guess` to Σ goals of all FINISHED matches; **20 split** `floor(20/n_tied)`, min 1 | `tournament:total_goals:<uid>` | `tournament` |
| **Highest match** | `score_highest_match_goals_guess()` | Final `FINISHED` | closest `highest_match_goals_guess` to `max(home+away)` over FINISHED; **15 split** | `tournament:highest_match:<uid>` | `tournament` |
| **Troublemaker** | `score_troublemaker()` | Final `FINISHED` | `player_prop_predictions(prop_key='troublemaker')` picks the (tied) max card‑weight player (Y=1, YR=1, R=2) → **15** (all ties win) | `tournament:troublemaker:<uid>` | `tournament` |
| **Group winner** | `score_group_winner(group)` | all 3 of a group's matches `FINISHED` (one‑shot, gated by `group_settlements`) | `group_winner_predictions.team_id` == actual winner (pts → GD → GF → team_id) → **5/group** (max 60) | `tournament:group_winner:<grp>:<uid>` | `tournament` |
| **First eliminated** | `score_first_eliminated()` | a team is first mathematically out of group top‑2 (one‑shot, gated by `first_elimination`) | `first_eliminated_team_id` == that team → **10** | `tournament:first_elim:<uid>` | `tournament` |

**Leaderboard roll‑up** (`league_standings` materialized view): per `(league, user)`
it sums `point_awards.points`, split into `match_points` / `bracket_points` /
`tournament_points` / `prop_points` by `prediction_type`, plus `total_points`.

> Note the **categorisation subtlety**: *troublemaker* is read from
> `player_prop_predictions` but awarded as `prediction_type='tournament'`, so it
> lands in **tournament_points**, not prop_points. Only the admin‑resolved player
> props (via `player_prop_resolutions`) are `prediction_type='prop'`. See §8‑e.

## 4. The lifecycle — when scorers run

`syncFixtures()` (cron `*/10 min`, or admin "Sync fixtures") does, in order:

1. Upsert matches (`onConflict: external_id`).
2. `score_match(id)` for each newly‑FINISHED match.
3. `score_bracket()` if any knockout match was seen.
4. `score_tournament()` **only once the Final is FINISHED** (chains into
   total‑goals / highest‑match / troublemaker sub‑scorers). Its top‑scorer block and all of `score_troublemaker()` are **gated on `all_match_details_synced()`** (migration `0016`) — they defer until no FINISHED match has `details_synced_at IS NULL`, so neither resolves on a partial drain backlog.
5. `backfill_team_group_letters()`.
6. `settle_group_stage_props()` → `score_first_eliminated()` + `score_group_winner()`
   per unsettled group.
7. `drainPendingMatchDetails(5)` → writes `player_goal_log` + `player_card_log`
   from per‑match details (this is the **only** feeder for top‑scorer & troublemaker).
8. `refresh_league_standings()`.

`syncScorers()` (cron daily 06:00) is the **detail‑drain backstop** (since #83): it
drains up to **8** pending FINISHED‑match details via `drainPendingMatchDetails()`
— spending no request on the `/scorers` list, so its full 10 req/min budget goes to
details — then re‑runs `score_tournament()` + `refresh_league_standings()`. This
accelerates the top‑scorer / troublemaker drain‑gate from step 4; it no longer just
logs the scorers list with no scoring effect (see §8‑b).

**Locks** (from the single `tournament` row): `round1_locked` (`now ≥ first_kickoff_at`)
blocks match / tournament / group‑winner / player‑prop writes; `round2_locked`
(`now ≥ knockout_start_at`) blocks bracket writes. Enforced by DB triggers **and**
the UI. Individual match picks also lock at each match's own `kickoff_at`.

## 5. Worked end‑to‑end example (the verified simulation)

### 5.1 Journey simulated

1. **Signup** — 4 users inserted into `auth.users`; the `handle_new_user` trigger
   auto‑created `profiles` with derived usernames.
2. **Onboarding** — `completeOnboarding` equivalent: each picks a username,
   `onboarded=true`; `athos` is admin/owner.
3. **League + invite** — `athos` creates league *The Three Musketeers*; `porthos`
   & `aramis` added; `dartagnan` joins via the real **`redeem_league_invite`** RPC
   (returned `ok=t, slug=musketeers`) then onboards.
4. **Catalog** — 48 ranked teams (codes/ranks from `fifa-rankings.ts`) across 12
   groups A–L; 96 players (a Forward + Defender per team).
5. **Round‑1 predictions** (pre‑lock) — 288 group 1X2 picks, 37 group‑winner picks,
   4 tournament‑prediction rows, 8 player‑prop picks.
6. **Round‑2 predictions** (pre‑lock) — 91 bracket picks (R32→W).
7. **Kickoff** — both locks moved to the past.
8. **Group stage** — 72 results, `score_match` ×72, `settle_group_stage_props`.
9. **Knockouts** — 32 results R32→Final, goals/cards logged, player prop resolved,
   `score_bracket` + `score_tournament`, `refresh_league_standings`.

### 5.2 The synthetic "truth" (deterministic, fully verifiable)

- Group winners = strongest seed of each group: **ARG, FRA, ESP, ENG, BRA, NED,
  POR, BEL, GER, CRO, COL, URU**.

## 8. Errors & observations catalogue (NOT fixed — per instructions)

Sentry was a no‑op here (`NEXT_PUBLIC_SENTRY_DSN` unset), so nothing shipped to
Sentry; the items below are what *would* surface, plus latent issues found by code
inspection during the run.

**Runtime / Sentry‑class (expected, sandbox‑only):**
- **`sync-fixtures` / `sync-scorers` 500** — football‑data.org unreachable + dummy
  token, and the service client points at a dummy Supabase URL. Handled
  (`{ok:false}` + `external_sync_log` + `Sentry.captureException` tag `cron`). Not a
  bug; an artifact of running without a backend.
- **Authenticated pages unreachable** — no Supabase Auth → every `(app)` route
  redirects to `/login`. Limitation, not a defect.

**Latent / design observations (real, worth a follow‑up):**
- **(a) `points_dark_horse()` is dead code.** Defined (=10) in `0002` but unused
  since dark horse became rank‑based in `0005`/`0014`. It's also *not* in
  `rules.ts`, so it can't drift the points‑sync check — but it's a footgun for
  anyone reading the SQL. Safe to delete in a future migration.
- **(b) `syncScorers()` detail‑drain lag — RESOLVED on current `main` (#83 / #85),
  after this simulation first ran.** Previously the daily `sync-scorers` cron only
  logged the scorers list and fed no scoring table, and the sole `player_goal_log` /
  `player_card_log` feeder was `drainPendingMatchDetails()` inside `sync-fixtures`
  (capped 5 matches/run) — so right after the Final, top‑scorer / troublemaker could
  resolve on a partial drain backlog. Migration `0016_gate_tournament_drain.sql` now
  gates `score_tournament()`'s top‑scorer block and all of `score_troublemaker()` on
  `all_match_details_synced()`, and `syncScorers()` was repurposed into a daily drain
  backstop (cap 8). Those two categories now defer until the drain completes and
  self‑heal — no wrong winner in the window. (Kept in this catalogue because the
  original run predated the fix; every other item here remains unfixed.)
- **(c) `score_first_eliminated()` is an approximation.** It flags a team as out
  when ≥2 group rivals exceed its max possible points, i.e. out of the **group**
  top‑2 — but WC 2026 also advances 8 best 3rd‑place teams, so "out of top‑2" ≠ "out
  of tournament." Documented in `0005`/migrations `CLAUDE.md`. (In‑sim it resolved
  cleanly to CZE.)
- **(d) Group‑stage props don't reconcile.** `score_group_winner` /
  `score_first_eliminated` are one‑shot (gated by `group_settlements` /
  `first_elimination`). If an already‑settled group result is corrected later, the
  `0014` reconcile rewrite does **not** re‑settle it (explicitly out‑of‑scope in
  `0014`). A post‑settlement correction would leave a stale group‑winner/first‑elim
  award.
- **(e) Troublemaker shows under *tournament_points*, not *prop_points*.** It comes
  from the `player_prop_predictions` table but is awarded `prediction_type='tournament'`.
  Intentional‑looking but easy to misread on the leaderboard breakdown.
- **(f) A `3RD`‑place bracket pick would score 0 but still create an award row.**
  The 3rd‑place match has a real `bracket_slot='3RD'` row, and `score_bracket` joins
  any slot; `points_bracket_slot('3RD')` returns 0, so a stray pick yields a
  0‑point `bracket` award. The UI never offers `3RD` as a slot, so this is only
  reachable via direct writes (the sim injected and then removed two such rows).
- **(g) Dark horse with a NULL `fifa_ranking` scores 0** even if the team reaches
  QF (e.g. a team that isn't in the seeded rank list). By design (the join requires
  `fifa_ranking is not null`).


## 10. Conclusion

The scoring system is **correct, complete, and idempotent**: all 16 point constants
match `rules.ts`, all 12 award categories produce exactly the expected points for a
fully‑specified tournament, and re‑running the scorers is a no‑op. CI is green. The
only failures observed are the expected infra errors from running without a live
Supabase/football‑data backend, plus a short list of latent/design observations
(§8) left **unfixed** as requested.
